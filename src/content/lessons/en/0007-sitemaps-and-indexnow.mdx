<p class="recap">Recap from <a href="/lessons/0006-the-js-rendering-gap.html">Lesson 0006</a>: we made sure your content is actually <em>in</em> the HTML the server sends. Now the page exists and renders. But an engine still has to <em>find out</em> it exists, or changed. That discovery step is its own pipeline stage — and it's the most automatable one in the whole course.</p>

<p class="lede">Two notification mechanisms, and the asymmetry is the lesson. A <b>sitemap</b> is <em>passive</em>: a list of your URLs the engine pulls when it feels like it — slow, but universal, Google included. <b>IndexNow</b> is <em>active</em>: you POST changed URLs and participating engines fetch within minutes, then share the ping with each other.<sup>[2]</sup> The catch: <b>Google doesn't participate in IndexNow</b> — it relies on its own crawl scheduling plus your sitemap.<sup>[3]</sup> So a real publishing pipeline fires both.</p>

<Callout variant="win" lead="Your win:">`sitemap_ping.py` <em>generates</em> a spec-valid `sitemap.xml` from a URL list, <em>validates</em> an existing one against the real sitemaps.org limits, and <em>builds</em> a correct IndexNow payload (key + keyLocation + urlList) — the exact two calls a publish hook should make.</Callout>

## Who listens to what

| Mechanism | Google | Bing / Yandex / Naver / Seznam | Speed |
|-----------|--------|--------------------------------|-------|
| **`sitemap.xml`** (passive pull) | yes | yes | slow — on the engine's schedule |
| **IndexNow** (active push) | no — ignores it<sup>[3]</sup> | yes — shared across all<sup>[2]</sup> | minutes |

Why a builder cares about the engines Google snubs: several of them feed <em>AI</em> answers (Bing's index sits behind Copilot and others), so an instant IndexNow push is partly an <span class="tag aeo">AEO</span> discovery move, not just classic SEO.

## The tool: build, validate, push

Both halves are pure data transforms — perfect for a publish hook, and offline-testable. The IndexNow side validates everything <em>before</em> it would ever hit the network:

```text
# IndexNow payload is built + checked offline; --send is the only network call
key matches  ^[A-Za-z0-9-]{8,128}$     # or it's rejected
all URLs share one host                # keyLocation must be on it too
<= 10,000 URLs per post
# sitemap side: well-formed, <=50k URLs, <=50MB, one host, valid lastmod
```

That generated `sitemap.xml` is plain spec-valid XML — one `<loc>` per URL, optional `<lastmod>`, all on one host:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://yoursite.com/new-page</loc>
    <lastmod>2026-06-22</lastmod>
  </url>
</urlset>
```

…and the IndexNow dry run prints the exact POST body before any `--send`:

```json
{
  "host": "yoursite.com",
  "key": "a1b2c3d4e5f6a7b8",
  "keyLocation": "https://yoursite.com/a1b2c3d4e5f6a7b8.txt",
  "urlList": ["https://yoursite.com/new-page"]
}
```

<Callout variant="gotcha" lead="Three pipeline traps.">(1) <b>IndexNow ≠ Google.</b> Pinging IndexNow and watching nothing happen in Google is expected — Google never took it.<sup>[3]</sup> Google's only push API is the <em>Indexing API</em>, limited to `JobPosting` and livestream pages; for everything else it's sitemap + crawl schedule, with a manual "Request indexing" in Search Console. (2) <b>The key file must actually be reachable</b> at `keyLocation` and contain the key, or every push is rejected — same trust model as `robots.txt`. (3) <b>A sitemap is discovery, not a ranking lever.</b> Listing a URL gets it <em>found</em>; it does nothing for whether it ranks or gets cited. Don't expect traffic from a sitemap alone.</Callout>

<p class="fineprint">Ceiling to know: `sitemap_ping.py` writes a single sitemap; past 50k URLs you need a <em>sitemap index</em> (a sitemap of sitemaps) — noted in the tool as the next upgrade. It defaults IndexNow to a dry run; `--send` is the one line that touches the network.</p>
