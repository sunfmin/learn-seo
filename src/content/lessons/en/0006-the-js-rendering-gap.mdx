<p class="recap">Recap from <a href="/lessons/0002-crawlable-vs-indexable.html">Lesson 0002</a>: you ran `crawl_audit.py` and it said "crawlable + indexable." But that tool fetches with `urllib` — it reads the <em>raw</em> response and never runs a line of JS. So does the page it blessed actually <em>contain</em> anything? This lesson finds the gap.</p>

<p class="lede">Google doesn't index your page in one shot. It crawls the <b>raw HTML</b> first, then — separately, later, when its resources allow — a headless Chrome <b>renders</b> the page, runs the JavaScript, and re-indexes whatever the JS produced.<sup>[1]</sup> Rendering is a <em>deferred queue</em>, not part of the crawl. If your content only exists after JS, it's invisible until that second wave — and to many bots, invisible forever.</p>

<Callout variant="win" lead="Your win:">run `render_gap.py` on a page's <em>raw</em> HTML vs its <em>rendered</em> DOM and get a per-signal diff — words, links, JSON-LD, headings — flagging exactly which ones exist <b>only after JavaScript</b>. That's the content a no-JS bot never sees and Google sees late.</Callout>

## Two waves, not one

Straight from Google's JavaScript-SEO doc: pages are processed in phases, and rendering sits in its own queue.<sup>[1]</sup>

<p>Google <em>does</em> run your JavaScript "using a recent version of Chrome, similar to how your browser renders pages."<sup>[2]</sup> The catch is <em>when</em>, and <em>who else doesn't</em>.</p>

## Your audit tool is a no-JS bot — and so are most AI crawlers

`crawl_audit.py` from 0002 is exactly the kind of fetcher that sees wave-1 only:

```python
# crawl_audit.py — the fetch is plain HTTP. No browser, no JS.
req = urllib.request.Request(url, headers={"User-Agent": "Googlebot"})
body = urllib.request.urlopen(req).read()   # <- raw HTML only
```

<p>So if your `<title>`, content, and canonical are injected by React after load, `crawl_audit.py` can't see them — and it might still print PASS off a near-empty shell. The same blind spot hits the AEO side hard: standalone AI crawlers commonly fetch raw HTML and <b>don't execute JavaScript</b>, and there's no settled standard for controlling them.<sup>[3]</sup> Client-side content that Googlebot eventually renders may never reach an answer engine at all.</p>

## The tool: diff the two versions

<p>You give `render_gap.py` two things — the <em>raw</em> HTML (it can fetch this, like 0002's tool) and the <em>rendered</em> DOM (you capture it: DevTools → Elements → right-click `<html>` → <em>Copy outerHTML</em>). It extracts the same signals from each and flags any that jump from ≈0 to a lot:</p>

```python
# share of each signal present WITHOUT JS
visible = raw[key] / rendered[key]          # 1.0 = fully server-rendered
js_dep  = rendered[key] > 0 and visible < 0.5   # needs JS to exist
# headline keys on visible TEXT: >=90% server-rendered, <=10% client-rendered
```

<Callout variant="do" lead="Do this now:">

1. Self-check (offline): `python3 tools/render_gap.py --demo`
2. Pick a page. Save its rendered DOM as `rendered.html` (DevTools → Copy outerHTML), then: `python3 tools/render_gap.py https://yoursite.com/page rendered.html`
3. Test a <b>known SPA</b> (a React/Vue app route) the same way — watch every signal flag <b>JS-DEP</b>. That's what a no-JS crawler sees.

<Callout variant="gotcha" lead="The JSON-LD trap: your perfect schema from 0003 can be invisible.">If you inject structured data through Google Tag Manager or client-side JS, it lives <em>only</em> in the rendered DOM — so `schema_tool.py` validates it, Googlebot may pick it up on wave 2, but a no-JS AI crawler never sees a single property.<sup>[3]</sup> Same story for client-rendered `<title>` and canonical: `crawl_audit.py` can wave a page through while the raw response is an empty shell. <b>Passing the crawl/index gate ≠ the content is actually there.</b> Put facts in the HTML the server sends.</Callout>

<p class="fineprint">Ceiling to know: `render_gap.py` doesn't run a browser — you supply the rendered DOM, the same way 0005's tracker takes engine output you captured. Its "visible words" counts text nodes, not layout; a fully JS-built page that Google renders fine will still flag here, and that's the point — it shows the <em>no-JS</em> view, which is one real audience, not a verdict on Googlebot.</p>
