<p class="recap">回顾 <a href="/zh/lessons/0002-crawlable-vs-indexable.html">第 0002 课</a>：你的页面现在已经可被 crawl（抓取）、可被 index（索引）。这让它<em>进入了</em>系统。结构化数据（structured data）是让它脱颖而出的第一根杠杆——也是第一个能<b>同时</b>服务两条流水线的东西。</p>

<p class="lede">结构化数据（structured data）是你嵌入页面、供机器读取的标记，它明确地说："这是一个 Product，价格是 29 USD，有现货。"不需要 NLP 去猜。Google 只推荐一种格式：<b>JSON-LD</b>——一个装着 JSON 的 `<script>` 块，与你可见的 HTML 解耦。<sup>[1]</sup></p>

<Callout variant="win" lead="你的收获：">为一个页面生成有效的 JSON-LD，并用 Google 的必填属性规则去 lint 它——在你发布<em>之前</em>就抓出"这赢不了 rich result（富结果）"，而不是几周后才在 Search Console 里发现。</Callout>

## 散文 vs. 带标签的事实

爬虫在散文里看到的是：*"我们的 Acme Widget Pro 现已发售，仅售二十九美元……"*——引擎必须去<b>推断</b>：那是个产品吗？是价格吗？哪种货币？有现货吗？也许对，也许不对。同样的事实写成 JSON-LD，则不留任何猜测空间：

```json
{
  "@type": "Product",
  "name": "Acme Widget Pro",
  "offers": {
    "price": "29.00",
    "priceCurrency": "USD",
    "availability": "InStock"
  }
}
```

## 必填 vs. 推荐

每种类型都有一些被 Google 标为 <Pill kind="req" />（少一个 → 完全没有 rich result）的属性，以及标为 <Pill kind="rec" />（跳过 → 仍然有效，只是不够丰富）的属性。分清哪个是哪个，就是这件事的全部——而这恰恰是 linter 所自动化的。

| 类型 | 它需要什么 |
|------|---------------|
| `FAQPage` | <Pill kind="req" /> `mainEntity` → 每个 `Question` 都需要 `name` + `acceptedAnswer.text` |
| `Product` | <Pill kind="req" /> `name`，以及在 `offers` 里：`price` + `priceCurrency` |
| `Article` | <Pill kind="rec" /> `headline`、`image`、`datePublished`、`author` |

## 这项技能：用脚本生成并 lint

在这个工作区里：`tools/schema_tool.py`——仅用标准库。它能为某个类型输出一段可复制粘贴的样例，并按上面的规则去 lint 任意 JSON-LD 文件。
