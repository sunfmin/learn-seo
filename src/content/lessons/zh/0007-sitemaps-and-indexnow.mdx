<p class="recap">回顾 <a href="/zh/lessons/0006-the-js-rendering-gap.html">第 0006 课</a>：我们确认了你的内容确实<em>存在于</em>服务器发送的 HTML 之中。现在页面存在了，也能渲染了。但引擎仍然必须<em>发现</em>它的存在，或它发生了变更。这一发现步骤本身就是管线中的一个阶段——而且是整门课里最容易自动化的一个。</p>

<p class="lede">两种通知机制，而它们之间的不对称正是本课的重点。<b>sitemap</b> 是<em>被动的</em>（passive）：一份你的 URL 列表，引擎想拉就拉——慢，但通用，连 Google 也包括在内。<b>IndexNow</b> 是<em>主动的</em>（active）：你 POST 变更过的 URL，参与的引擎会在几分钟内抓取，然后彼此之间共享这次 ping。<sup>[2]</sup> 问题在于：<b>Google 不参与 IndexNow</b>——它依赖自己的抓取调度加上你的 sitemap。<sup>[3]</sup> 所以一条真正的发布管线（publish pipeline）两者都会触发。</p>

<Callout variant="win" lead="你的收获：">`sitemap_ping.py` 能从一份 URL 列表<em>生成</em>一个符合规范的 `sitemap.xml`，对照 sitemaps.org 的真实上限<em>校验</em>一个现有文件，并<em>构建</em>一个正确的 IndexNow payload（key + keyLocation + urlList）——正是一个发布钩子应该发起的那两次调用。</Callout>

## 谁听谁的

| 机制 | Google | Bing / Yandex / Naver / Seznam | 速度 |
|-----------|--------|--------------------------------|-------|
| **`sitemap.xml`**（被动拉取） | 是 | 是 | 慢——按引擎自己的节奏 |
| **IndexNow**（主动推送） | 否——忽略它<sup>[3]</sup> | 是——所有引擎共享<sup>[2]</sup> | 几分钟 |

为什么开发者要关心那些被 Google 冷落的引擎：其中好几个为 <em>AI</em> 答案供料（Bing 的索引就支撑着 Copilot 等产品），所以一次即时的 IndexNow 推送在某种程度上是一步 <span class="tag aeo">AEO</span> 发现动作，而不只是经典 SEO。

## 工具：构建、校验、推送

这两半都是纯数据转换——非常适合放进一个发布钩子里，而且可以离线测试。IndexNow 这一侧会在任何数据真正触网<em>之前</em>就把一切校验完毕：

```text
# IndexNow payload is built + checked offline; --send is the only network call
key matches  ^[A-Za-z0-9-]{8,128}$     # or it's rejected
all URLs share one host                # keyLocation must be on it too
<= 10,000 URLs per post
# sitemap side: well-formed, <=50k URLs, <=50MB, one host, valid lastmod
```

生成的那个 `sitemap.xml` 是纯粹符合规范的 XML——每个 URL 一个 `<loc>`，可选的 `<lastmod>`，全都在同一个 host 上：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://yoursite.com/new-page</loc>
    <lastmod>2026-06-22</lastmod>
  </url>
</urlset>
```

……而 IndexNow 的试运行（dry run）会在任何 `--send` 之前打印出准确的 POST 请求体：

```json
{
  "host": "yoursite.com",
  "key": "a1b2c3d4e5f6a7b8",
  "keyLocation": "https://yoursite.com/a1b2c3d4e5f6a7b8.txt",
  "urlList": ["https://yoursite.com/new-page"]
}
```

<Callout variant="do" lead="现在就做：">

1. 自检（离线）：`python3 tools/sitemap_ping.py --demo`
2. 做一个 `urls.txt`（每行一个 URL，可选地加一个 tab + lastmod），生成并校验：`python3 tools/sitemap_ping.py gen urls.txt > sitemap.xml`，然后 `python3 tools/sitemap_ping.py check sitemap.xml`
3. 构建一次 IndexNow 推送（试运行——会打印出 key 文件那一步以及准确的 POST 请求体）：`python3 tools/sitemap_ping.py indexnow https://yoursite.com/new-page --key <your-key>`

</Callout>

<Callout variant="gotcha" lead="三个管线陷阱。">（1）<b>IndexNow ≠ Google。</b>向 IndexNow 推送后却看到 Google 那边毫无动静，这是预料之中的——Google 从未采纳它。<sup>[3]</sup> Google 唯一的推送 API 是 <em>Indexing API</em>，仅限 `JobPosting` 和直播页面；其他一切都走 sitemap + 抓取调度，外加 Search Console 里手动的"请求编入索引"（Request indexing）。（2）<b>key 文件必须真的可以访问到</b>，位于 `keyLocation` 处并且包含那个 key，否则每次推送都会被拒绝——和 `robots.txt` 是同一套信任模型。（3）<b>sitemap 是发现手段，不是排名杠杆。</b>列出一个 URL 能让它被<em>找到</em>；至于它能否排名或被引用，sitemap 起不到任何作用。别指望单靠 sitemap 带来流量。</Callout>

<p class="fineprint">需要知道的上限：`sitemap_ping.py` 写的是单个 sitemap；超过 5 万个 URL 时你就需要一个 <em>sitemap index</em>（一份关于多个 sitemap 的 sitemap）——工具里把它标注为下一步升级项。它默认让 IndexNow 走试运行；`--send` 是唯一会触网的那一行。</p>
