<p class="recap">回顾 <a href="/zh/lessons/0002-crawlable-vs-indexable.html">第 0002 课</a>：你运行了 `crawl_audit.py`，它说“可抓取 + 可索引”。但那个工具是用 `urllib` 抓取的——它读取的是<em>原始</em>响应，从不运行一行 JS。那么它放行的那个页面，究竟<em>含有</em>任何东西吗？这一课就来找出这道缺口。</p>

<p class="lede">Google 并不是一次性就把你的页面索引（index）了。它先抓取（crawl）<b>原始 HTML（raw HTML）</b>，然后——单独地、在之后、在它的资源允许时——一个 headless Chrome 会<b>渲染（render）</b>这个页面，运行 JavaScript，再把 JS 产出的东西重新索引。<sup>[1]</sup>渲染是一个<em>延迟队列</em>，不是抓取的一部分。如果你的内容只在 JS 之后才存在，那么在第二波到来之前它都是不可见的——而对很多 bot 来说，则是永远不可见。</p>

<Callout variant="win" lead="你的收获：">在一个页面的<em>原始</em> HTML 和它的<em>渲染后的 DOM（rendered DOM）</em>上运行 `render_gap.py`，得到一份逐信号的差异对比——词数、链接、JSON-LD、标题——精确标出哪些信号<b>只在 JavaScript 之后</b>才存在。那正是一个不跑 JS 的（no-JS）bot 永远看不到、而 Google 看得很晚的内容。</Callout>

<p>Google <em>确实</em>会“使用近期版本的 Chrome，以类似浏览器渲染页面的方式”运行你的 JavaScript。<sup>[2]</sup>问题在于<em>什么时候</em>，以及<em>还有谁不运行</em>。</p>

## 你的审计工具是个不跑 JS 的 bot——大多数 AI 爬虫也是

第 0002 课里的 `crawl_audit.py`，正是那种只看得到第一波的抓取器：

```python
# crawl_audit.py — the fetch is plain HTTP. No browser, no JS.
req = urllib.request.Request(url, headers={"User-Agent": "Googlebot"})
body = urllib.request.urlopen(req).read()   # <- raw HTML only
```

<p>所以如果你的 `<title>`、内容和 canonical 是 React 在加载之后注入的，`crawl_audit.py` 就看不到它们——而它可能仍然会对着一个近乎空壳的页面打出 PASS。同样的盲区也重重地打在 AEO 这一侧：独立的 AI 爬虫通常抓取原始 HTML（raw HTML）并且<b>不执行 JavaScript</b>，而且控制它们也没有定型的标准。<sup>[3]</sup>那些 Googlebot 最终会渲染的客户端内容，可能根本到不了答案引擎（answer engine）那里。</p>

## 工具：对比两个版本

<p>你给 `render_gap.py` 两样东西——<em>原始</em> HTML（它可以自己抓，就像第 0002 课的工具那样）和<em>渲染后的 DOM（rendered DOM）</em>（这个由你来捕获：DevTools → Elements → 右键点 `<html>` → <em>Copy outerHTML</em>）。它从两者中提取相同的信号，并标记出任何从 ≈0 跳到很多的信号：</p>

```python
# share of each signal present WITHOUT JS
visible = raw[key] / rendered[key]          # 1.0 = fully server-rendered
js_dep  = rendered[key] > 0 and visible < 0.5   # needs JS to exist
# headline keys on visible TEXT: >=90% server-rendered, <=10% client-rendered
```

<Callout variant="do" lead="现在就动手：">

1. 自检（离线）：`python3 tools/render_gap.py --demo`
2. 挑一个页面。把它渲染后的 DOM 存为 `rendered.html`（DevTools → Copy outerHTML），然后：`python3 tools/render_gap.py https://yoursite.com/page rendered.html`
3. 用同样的方式测一个<b>已知的 SPA</b>（一个 React/Vue 应用的路由）——看着每个信号都被标成 <b>JS-DEP</b>。那就是一个不跑 JS 的爬虫看到的东西。

</Callout>

<Callout variant="gotcha" lead="JSON-LD 陷阱：你在第 0003 课里那份完美的 schema 可能是不可见的。">如果你是通过 Google Tag Manager 或客户端 JS 注入结构化数据，那它<em>只</em>存在于渲染后的 DOM 里——所以 `schema_tool.py` 能验证它，Googlebot 可能在第二波时拾取它，但一个不跑 JS 的 AI 爬虫一个属性都看不到。<sup>[3]</sup>客户端渲染的 `<title>` 和 canonical 也是同样的故事：`crawl_audit.py` 可以在原始响应是空壳的情况下放一个页面过关。<b>通过抓取/索引这道关卡 ≠ 内容真的在那里。</b>把事实放进服务器发出的 HTML 里。</Callout>

<p class="fineprint">要知道的上限：`render_gap.py` 不运行浏览器——渲染后的 DOM 由你提供，就像第 0005 课的追踪器拿的是你捕获的引擎输出一样。它的“visible words”统计的是文本节点，不是布局；一个完全由 JS 构建、Google 渲染起来毫无问题的页面，在这里仍然会被标记，而这正是重点——它展示的是<em>不跑 JS 的</em>视角，那是一个真实存在的受众，而不是对 Googlebot 的最终裁决。</p>
